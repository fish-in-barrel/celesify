# .instructions.md — GitHub Copilot Context

This file provides GitHub Copilot / Copilot Chat with project context, conventions, and guidance for working on celesify.

## Project Overview

**celesify**: A machine learning pipeline to classify stellar objects (stars, galaxies, quasars) from the SDSS17 dataset using scikit-learn Random Forests.

- **Dataset**: 100,000 objects, 17 photometric features, 3 classes
- **Goal**: Measure RF accuracy under default and tuned hyperparameters, extract feature importance
- **Architecture**: Docker Compose with 3 services (preprocessing → training → Streamlit dashboard)
- **Status**: Fully functional; training module refactored (April 2026) into 6 modular, well-documented files

## Code Style & Conventions

### Python Standards
- Python 3.11+, PEP 8 compliant
- Use `from __future__ import annotations` at top of all modules for forward compatibility
- Comprehensive type hints on all functions (`def func(x: int, y: str) -> dict:`)
- Docstrings for every public function, class, and module (Google/NumPy hybrid format)
- Keep functions pure and single-responsibility where possible

### Random State
- **Always** use `random_state=42` for reproducibility (from `celesify.core.constants.RANDOM_STATE`)
- Applied to: data splits, cross-validation, model initialization, any stochastic operation

### Logging
- Use `celesify.core.logging.log(SERVICE, message)` for all logging (not `print()`, not `logging.getLogger()`)
- `SERVICE` is typically module name (e.g., `"training"`, `"preprocessing"`)
- Log at key checkpoints: startup, phase transitions, artifacts written

### Class Encoding
- Fixed mapping: `STAR=0`, `GALAXY=1`, `QSO=2` (from `celesify.core.constants.CLASS_ENCODING`)
- Always use `CLASS_LABEL_ORDER = [0, 1, 2]` consistently across eval metrics

### Error Handling
- Data validation early and clearly (check for required columns, shapes before processing)
- ONNX export is **non-fatal** — if conversion fails, log and continue (joblib fallback always available)
- Provide clear error messages with actionable next steps

## Training Module Architecture (April 2026 Refactoring)

The training module was refactored from a 650+ line monolith into 6 focused modules:

### Module Breakdown

| Module | Lines | Responsibility | Key Functions |
|--------|-------|-----------------|---|
| `config.py` | 90 | Config mgmt | `TrainingConfig` class (reads env vars) |
| `data_handling.py` | 360 | Data I/O | Load datasets, extract features, subsample, check imbalance |
| `model_training.py` | 180 | Training | Baseline RF, RandomizedSearchCV |
| `evaluation.py` | 360 | Metrics | Evaluate, extract importance, format for export |
| `export.py` | 170 | Artifacts | Save JSON/joblib/ONNX with error handling |
| `pipeline.py` | 300 | Orchestration | `run()` entry point, phase coordination |

**See [TRAINING_MODULE_ARCHITECTURE.md](TRAINING_MODULE_ARCHITECTURE.md) for comprehensive technical guide.**

### Configuration (Environment Variables)

Supported in `TrainingConfig`:
- `TRAINING_N_ITER` — RandomizedSearchCV iterations (default: 20)
- `TRAINING_CV_SPLITS` — CV folds (default: 5)
- `TRAINING_N_JOBS` — Parallel jobs (default: -1)
- `TRAINING_MAX_TRAIN_ROWS` — Subsample limit for quick validation (default: 0 = no limit)

Example quick validation run:
```bash
TRAINING_N_ITER=1 TRAINING_CV_SPLITS=2 TRAINING_MAX_TRAIN_ROWS=3000 \
  python -m celesify.training.main
```

### Hyperparameter Search Space

```python
n_estimators: [100, 200, 300, 500]
max_depth: [None, 10, 20, 30]
min_samples_split: [2, 5, 10]
max_features: ["sqrt", "log2", 0.3]
class_weight: [None, "balanced"]  # if imbalance detected
```

**Scoring**: `f1_macro` (macro-averaged F1, required under class imbalance)  
**CV**: `StratifiedKFold` (preserves class distribution)

## Data Pipeline

### Preprocessing Stage
- Inputs: `data/raw/star_classification.csv` (SDSS17 from Kaggle)
- Outputs: `outputs/processed/train.parquet`, `outputs/processed/test.parquet`, `outputs/processed/preprocessing_report.json`
- Features: `[alpha, delta, u, g, r, i, z, redshift]` (after dropping non-informative columns)
- Split: 80/20 train/test, stratified by class
- Imbalance handling: Class imbalance checked; if ratio > 2.0, recommend `class_weight='balanced'`

### Training Stage
- Inputs: Parquet files from preprocessing
- Outputs: `baseline_metrics.json`, `tuned_metrics.json`, `best_params.json`, `feature_importance.json`, `model.joblib`, `model.onnx` (if successful)
- Two model variants:
  1. **Baseline**: Default RF (n_estimators=100) on clean features
  2. **Tuned**: Best hyperparameters from search on engineered features

### Metrics Exported

**Baseline & Tuned Metrics**:
```json
{
  "accuracy": <float>,
  "f1_macro": <float>,
  "confusion_matrix": [[...], [...], [...]],
  "per_class_metrics": {
    "0": {"precision": ..., "recall": ..., "f1_score": ..., "support": ...},
    ...
  }
}
```

**Feature Importance** (MDI):
```json
[
  {"feature": "redshift", "importance": 0.312},
  {"feature": "u", "importance": 0.245},
  ...
]
```

## Streamlit Dashboard (April 2026)

The dashboard provides three integrated views for exploring data, evaluating models, and running inference. All data loads from the shared `outputs/` volume at startup.

### Architecture

- **Model Variant Selection**: Sidebar radio selector to choose between:
  - **Baseline (default)**: Original RF with 8 base features only
  - **Clean Tuned**: Tuned RF with cleaned original features
  - **Tuned (engineered)**: Tuned RF with engineered features (colors, band statistics, interactions)
- **Dynamic Metrics Loading**: Loads corresponding metrics file (baseline_metrics.json, clean_tuned_metrics.json, or tuned_metrics.json) based on selected variant
- **All Views Update**: Data Explorer, Model Evaluation, and Upload & Infer all reflect the selected model's feature schema and metrics

### View 1: Data Explorer (`page_data_explorer.py`)

**Context-Aware Feature Defaults**:
- **Cleaned baseline mode**: Defaults show only original features (alpha, delta, u, g, r, i, z, redshift)
- **Engineered mode**: Defaults show mix of original + engineered features (color differences, band stats)

**Sections**:
1. **Preprocessing Comparison**: Metrics on rows removed, features added by engineering
2. **Distributions**: Class bar chart + per-feature histograms (user-selectable)
3. **Correlation**: Photometric band correlations (with optional redshift)
4. **Univariate Exploration**: Boxplots by class for selected features
5. **Multivariate Analysis**: Pair plots showing 2D feature interactions

**Implementation Notes**:
- Samples training or test data based on availability; allows user to control sample size for plotting
- All plots use `random_state=42` for reproducibility
- Feature selector multiselect respects dataset variant (cleaned vs engineered)

### View 2: Model Evaluation (`page_performance_metrics.py`)

**Displays**:
1. **Confusion Matrix Heatmap**: For selected model variant
2. **Per-Class Metrics Table**: Precision, recall, F1, support (class IDs 0, 1, 2 → STAR, GALAXY, QSO)
3. **Feature Importance Rankings**: MDI scores from trained RF (top 15 features by default)
4. **Best Hyperparameters**: Displays tuned model's optimal params (n_estimators, max_depth, etc.)

**Data Source**: All metrics loaded from JSON at startup; no runtime computation.

### View 3: Upload & Infer (`page_upload_infer.py`)

**Input Modes**:
1. **Manual Input**: 8-field form for original parameters only:
   - alpha, delta, u, g, r, i, z, redshift
2. **CSV Upload**: Upload CSV with columns matching original features

**Feature Engineering**: 
- Automatically calculates engineered features from original parameters:
  - 9 color differences (u-g, g-r, r-i, i-z, etc.)
  - 5 band statistics (mean, std, min, max, range across u,g,r,i,z)
  - 5 redshift-color interactions (redshift × each major color)

**Outputs**:
- Predicted class label (STAR, GALAXY, or QSO)
- Per-class probability bar chart
- Full prediction dataframe with probabilities and predictions

**Implementation Notes**:
- User sees only original feature inputs; engineered features computed transparently
- Handles missing/malformed data gracefully with clear error messages
- Batch inference available via CSV upload

## Conventions for Extensions

### Adding a New Function to Training Module

1. Place in appropriate file (data_handling, model_training, evaluation, or export)
2. Add comprehensive docstring with Args, Returns, raises (if applicable)
3. Use full type hints: `def func(x: pd.DataFrame, n: int) -> dict[str, Any]:`
4. Use `RANDOM_STATE` for any stochastic operation
5. Log at entry/exit if significant work: `log(SERVICE, "...")`
6. Add to module __all__ if public API

### Adding New Configuration

1. Add to `TrainingConfig.__init__()` in `config.py`
2. Add environment variable support via `_get_int_env()`
3. Log in `_log_summary()` for visibility
4. Document in TRAINING_QUICK_REFERENCE.md

### Modifying Metrics Computation

1. Edit `evaluation.evaluate_model()` for core metrics
2. Update corresponding `format_*_metrics()` function
3. Ensure per-class metrics use `CLASS_LABEL_ORDER` consistently
4. Test with sample data before committing

### Extending the Streamlit Dashboard

1. **Adding a new view**: Create new `page_*.py` module in `streamlit_app/`, export `render_*()` function
2. **Model variant support**: Use `get_available_model_variants()` from `common.py` to discover models
3. **Feature engineering**: Use `engineer_features()` from `page_upload_infer.py` for automatic feature calculations
4. **Context-aware defaults**: Check `dataset_variant` (cleaned vs engineered) before setting feature selector defaults
5. **Caching**: Use `@st.cache_data` for JSON/Parquet I/O, `@st.cache_resource` for model loading
6. **Error handling**: Catch `FileNotFoundError` gracefully and show user-friendly messages via `st.warning()` or `st.error()`
7. **Feature lists**: Keep original feature names in `ORIGINAL_FEATURES` dict; use engineering functions for computed features
